# 公共层开发与维护指南
## 1. 公共层的定位
公共层对应目录：

`Code/03_Common`

这一层的核心目标是：

1. 放纯算法
2. 放通用工具
3. 放跨模块可复用的小型基础能力

它和 `Task/Module/Driver` 的最大区别是：

**公共层不应该依赖具体硬件。**

## 2. 当前公共层包含什么
当前主要包含两类内容：

1. PID 控制算法
2. 通用工具函数

对应文件大致是：

| 文件 | 作用 |
| --- | --- |
| `PID/pid_config.h` | PID 默认参数配置 |
| `PID/pid_pos.*` | 位置式 PID |
| `PID/pid_inc.*` | 增量式 PID |
| `PID/pid_multi.*` | 多环/串级 PID 组合框架 |
| `Src/common_checksum.c` | 通用校验 |
| `Src/common_str.c` | 数值转字符串 |

## 3. 公共层设计原则
### 3.1 纯算法优先

公共层里的代码应尽量做到：

1. 不依赖具体 GPIO
2. 不依赖具体 UART
3. 不依赖任务调度
4. 不依赖具体项目状态机

### 3.2 接口尽量稳定

公共层一旦稳定，复用价值就很高。  
所以除非必要，不要频繁破坏接口语义。

### 3.3 参数集中管理

像 PID 这样的算法，不应把参数散落在任务文件里。  
当前工程把默认参数收敛到 `pid_config.h`，这是正确方向。

## 4. `pid_config.h` 详解
### 4.1 它的作用

`pid_config.h` 是当前 PID 默认参数集中入口。  
它的意义不是“把数字放一起”，而是：

1. 让调参入口清楚
2. 避免参数散落
3. 让任务层和算法层共享统一参数基线

### 4.2 当前关键参数

#### 目标值

| 参数 | 当前值 |
| --- | --- |
| `MOTOR_TARGET_SPEED` | `40` |
| `MOTOR_TARGET_ERROR` | `0` |

#### 位置环参数

| 参数 | 当前值 |
| --- | --- |
| `MOTOR_POS_KP` | `0.0025` |
| `MOTOR_POS_KI` | `0.0` |
| `MOTOR_POS_KD` | `0.0` |
| `MOTOR_POS_OUTPUT_MAX` | `0.5` |
| `MOTOR_POS_INTEGRAL_MAX` | `5000.0` |

#### 速度环参数

| 参数 | 当前值 |
| --- | --- |
| `MOTOR_SPEED_KP` | `8.0` |
| `MOTOR_SPEED_KI` | `3.0` |
| `MOTOR_SPEED_KD` | `0.2` |

### 4.3 修改这些参数时必须注意

1. 改参数不是只看编译通过。
2. 必须重新做联调验证。
3. 如果参数语义发生变化，要同步更新文档。

## 5. `pid_pos` 详解
### 5.1 适合什么场景

位置式 PID 适合：

1. 绝对偏差控制
2. 需要清楚看出 `P/I/D` 组成的场景
3. 位置、角度、轨迹偏差等控制

### 5.2 当前工程里的使用

当前主要用于：

1. 底盘直线模式中的位置纠偏
2. 步进任务中的视觉误差收敛控制

### 5.3 它的重要特性

1. 支持积分限幅
2. 支持输出限幅
3. 支持复位内部状态

这对任务切换很重要，因为否则切模式时历史积分很容易污染新状态。

## 6. `pid_inc` 详解
### 6.1 适合什么场景

增量式 PID 更适合：

1. 速度控制
2. 需要输出平滑累加的场景

### 6.2 当前工程里的使用

当前主要用于：

1. 底盘左右轮速度环

### 6.3 为什么在当前工程里合理

因为底盘电机速度环本质上更适合增量式思路，输出变化更平滑，也更符合当前任务调节方式。

## 7. `pid_multi` 详解
### 7.1 它的定位

`pid_multi` 不是当前任务主路径里最核心的算法，但它体现了公共层的扩展方向：

1. 支持串级/多环结构
2. 支持外环和内环组合
3. 可复用于后续更复杂控制链路

### 7.2 为什么保留它有价值

因为当前工程已经有：

1. 位置外环
2. 速度内环

这种多环结构的雏形。  
把组合能力保留在公共层，是后续扩展的正确方向。

## 8. `common_checksum` 详解
### 8.1 它的作用

`common_checksum_xor_u8` 用于做字节流 XOR 校验。

### 8.2 当前工程里它服务谁

当前最典型是：

1. `mod_k230`

### 8.3 为什么这类函数应该放公共层

因为它：

1. 不依赖具体协议模块
2. 不依赖硬件
3. 可以被多个协议复用

## 9. `common_str` 详解
### 9.1 它的作用

`common_str` 提供轻量的数值转字符串能力，避免在嵌入式环境里过度依赖 `sprintf`。

### 9.2 当前工程里它服务谁

当前最典型是：

1. `mod_vofa`
2. OLED 相关的数字文本组织

### 9.3 为什么要保留这类工具

因为在嵌入式项目里：

1. 格式化输出很常见
2. 标准库格式化有时过重
3. 可控的轻量工具函数更适合长期维护

## 10. 公共层和其他层的边界
### 10.1 公共层和驱动层的边界

公共层不应读取外设，也不应写 GPIO。  
一旦开始依赖 `HAL`，它就不再是纯公共层了。

### 10.2 公共层和模块层的边界

公共层不应该知道：

1. `K230`
2. `VOFA`
3. `MG513X`
4. 灰度模块

这些都是项目设备语义，应该留在模块层。

### 10.3 公共层和任务层的边界

公共层不应该知道：

1. 当前任务周期
2. 当前状态机
3. 哪个按键切什么模式

这些都属于任务层。

## 11. 新增公共算法时的建议
后续新增算法或工具时，建议先判断：

1. 它是否和具体硬件无关
2. 它是否可能被多个模块复用
3. 它是否值得作为独立组件存在

如果答案基本是“是”，再考虑放进 `Common`。

## 12. 公共层最容易犯的错误
### 错误 1：把项目业务语义塞进公共层

这会让公共层失去复用价值。

### 错误 2：参数散落

尤其是 PID，一旦参数散落到多个任务文件里，后续调参会变得非常混乱。

### 错误 3：工具函数接口语义不稳定

公共层变化影响面很大，所以改接口前要比其他层更谨慎。

## 13. 公共层维护检查清单
每次修改公共层后，建议检查：

1. 是否引入了不必要的硬件依赖
2. 参数是否仍集中管理
3. 修改是否影响现有模块调用
4. 调整后的算法是否重新验证
5. 文档是否同步

## 14. 这份文档什么时候必须更新
只要下面内容变化，就必须更新：

1. PID 参数组织方式变化
2. 新增或删除算法组件
3. 公共工具接口变化
4. 适用场景变化
