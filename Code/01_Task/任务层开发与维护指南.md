# 任务层开发与维护指南
## 1. 这份文档的作用
这份文档专门说明 `Code/01_Task` 这一层在当前工程里到底负责什么、已经实现了什么、以后应该怎么维护。

首次接触本工程时，建议先看：

1. [../../README.md](../../README.md)
2. [../../docs/06_系统启动流程与运行逻辑总览.md](../../docs/06_系统启动流程与运行逻辑总览.md)

看完系统总览后，再回来看这份文档，会更容易理解。

## 2. 任务层在整个工程中的位置
本工程采用：

`Task -> Module -> Driver -> HAL`

任务层是最靠近“系统行为”的一层。  
它不负责底层引脚细节，也不负责最小驱动实现，而是负责：

1. 任务调度
2. 状态机
3. 时序
4. 多模块协作
5. 同步资源消费与分发

可将其理解为：

1. `Driver` 提供原子硬件能力
2. `Module` 提供设备语义
3. `Task` 把这些语义组织成“一个真的会工作的系统”

## 3. 任务层不应该做什么
为了保持当前结构可维护，任务层不应该承担下面这些职责：

1. 不应该直接写 HAL 外设寄存器或 `HAL_GPIO_WritePin` 之类底层调用。
2. 不应该把引脚映射散落在多个任务文件里。
3. 不应该在多个任务里同时修改同一份核心状态机。
4. 不应该把“能抽到 Module 的逻辑”继续堆在任务层。

尤其是下面这条最重要：

**任务层不要绕过模块层直接操作硬件。**

## 4. 当前目录结构与文件职责
当前 `Code/01_Task` 主要文件如下：

| 文件 | 作用 |
| --- | --- |
| `Inc/task_init.h` + `Src/task_init.c` | 系统装配入口，完成模块绑定和初始化 |
| `Inc/task_key.h` + `Src/task_key.c` | 按键扫描与业务信号量分发 |
| `Inc/task_gpio.h` + `Src/task_gpio.c` | LED/蜂鸣器/激光反馈仲裁 |
| `Inc/task_oled.h` + `Src/task_oled.c` | OLED 页面刷新与电压显示 |
| `Inc/task_dcc.h` + `Src/task_dcc.c` | 底盘 DCC 状态机与控制 |
| `Inc/task_stepper.h` + `Src/task_stepper.c` | 双轴步进控制任务 |
| `Inc/task_test.h` + `Src/task_test.c` | 测试预留任务 |

此外，任务本身的创建不在 `Code/01_Task`，而在：

1. `Core/Src/freertos.c`

维护任务层时，建议形成以下阅读与排查习惯：

1. 看任务逻辑，去 `Code/01_Task`
2. 看任务创建、优先级、栈和同步对象，去 `freertos.c`

## 6. 任务层最关键的机制：InitTask + 启动闸门
### 6.1 为什么要有 InitTask

本工程不是在 `main.c` 里把所有模块都初始化完，而是专门用一个 `InitTask` 做系统装配。  
它负责：

1. 建立默认上下文
2. 注入绑定表
3. 初始化基础模块
4. 绑定通信模块
5. 准备双轴步进通道
6. 释放启动闸门
7. 自删除

### 6.2 `task_wait_init_done()` 的作用

每个业务任务在主循环之前都先调用：

`task_wait_init_done()`

这个函数背后的含义是：

1. 业务任务不能抢跑
2. 只有等 `InitTask` 把系统装配完，其他任务才开始真正工作

这是当前工程稳定性的关键之一。  
后续新增任务时，默认也应先接入这一启动闸门。

## 7. `task_init.c` 详解
### 7.1 这是整个系统的装配中心

`task_init.c` 是任务层里最重要的文件之一。  
它负责维护一组非常关键的“绑定表”：

1. LED 绑定表
2. Relay 绑定表
3. Key 绑定表
4. Motor 绑定表
5. Sensor 绑定表
6. Battery 绑定参数

### 7.2 当前它绑定了什么

当前已经明确绑定了：

1. 红绿黄 3 个 LED
2. 激光和蜂鸣器 2 个继电器
3. 3 个按键
4. 左右两个直流电机
5. 12 路灰度传感器
6. 电池采样 ADC
7. VOFA 串口
8. K230 串口
9. 双轴步进通道

### 7.3 为什么以后改引脚时一定要看它

因为对当前工程来说，真正决定“某个模块绑定到哪个引脚/哪个句柄”的地方不是任务主循环，而是这里。  
因此一旦修改：

1. UART 实例
2. TIM 通道
3. GPIO 口线
4. ADC 通道

都必须回来看 `task_init.c` 的绑定表是否要同步修改。

## 8. `task_key.c` 详解
### 8.1 它的职责

`KeyTask` 的职责非常单纯：

1. 周期扫描按键
2. 把模块层事件映射为业务信号量
3. 给 `GpioTask` 提供按键反馈触发

### 8.2 当前扫描周期

当前周期由：

`TASK_KEY_PERIOD_MS = MOD_KEY_SCAN_PERIOD_MS = 10ms`

### 8.3 当前事件映射

当前映射关系是：

1. `MOD_KEY_EVENT_2_CLICK -> Sem_DccHandle`
2. `MOD_KEY_EVENT_3_CLICK -> Sem_TaskChangeHandle`
3. `MOD_KEY_EVENT_3_DOUBLE_CLICK -> Sem_ReadyToggleHandle`

同时，任意有效事件都会释放：

`Sem_RedLEDHandle`

### 8.4 为什么它不直接改 DCC 状态

因为当前工程坚持“单一状态写入者”原则。  
按键任务只是事件源，真正改 `DCC` 状态的是 `DccTask`。

这使得状态流转更可控，也更容易排查。

## 9. `task_gpio.c` 详解
### 9.1 它的职责

`GpioTask` 是所有反馈输出的统一仲裁者。  
当前管理：

1. 红灯
2. 绿灯
3. 黄灯
4. 蜂鸣器继电器
5. 激光继电器

### 9.2 当前主要节奏参数

| 参数 | 当前值 |
| --- | --- |
| 主循环周期 | `20ms` |
| 黄灯短闪 | `120ms` |
| 按键短鸣 | `90ms` |
| ON 态绿灯闪烁周期 | `300ms` |
| STOP 态红灯闪烁周期 | `250ms` |
| STOP 态蜂鸣开启 | `90ms` |
| STOP 态蜂鸣关闭 | `210ms` |

### 9.3 当前输出策略

1. 任意有效按键都会触发黄灯短闪和短鸣。
2. `DCC ON` 时，激光继电器吸合。
3. `DCC ON` 时，绿灯闪烁。
4. `DCC STOP` 时，红灯闪烁并进行节奏蜂鸣。
5. 其他状态下，状态灯和继电器回到安全态。

### 9.4 维护规则

后续如果新增灯效、提示音或报警输出，优先考虑集中放到 `GpioTask`。  
不要在别的任务里各自写 GPIO，否则状态会很快打架。

## 10. `task_oled.c` 详解
### 10.1 它的职责

`OledTask` 负责：

1. 周期采样电池电压
2. 周期刷新页面
3. 显示模式和运行状态

### 10.2 当前显示时基

| 项目 | 当前值 |
| --- | --- |
| 页面刷新周期 | `500ms` |
| 电压采样周期 | `10s` |
| 空闲延时 | `50 tick` |

### 10.3 当前页面内容

当前页面主要显示：

1. `voltage`
2. `mode`
3. `task:ON/OFF/PREP/STOP`

### 10.4 为什么它把采样和刷新分开

因为：

1. 电压不需要高频采样
2. OLED 刷新也没必要跟采样同频
3. 分开以后 I2C 和 ADC 压力更合理

## 11. `task_dcc.c` 详解
### 11.1 它是底盘状态机唯一写入者

这句话非常重要。  
当前工程中，`DCC` 的模式和运行态，只允许 `DccTask` 改写。

这保证了：

1. 状态切换逻辑集中
2. 保护逻辑集中
3. 排查时不会出现多个任务互相覆盖状态

### 11.2 当前维护的两组状态

#### 模式

1. `TASK_DCC_MODE_IDLE`
2. `TASK_DCC_MODE_STRAIGHT`
3. `TASK_DCC_MODE_TRACK`

#### 运行态

1. `TASK_DCC_RUN_OFF`
2. `TASK_DCC_RUN_PREPARE`
3. `TASK_DCC_RUN_ON`
4. `TASK_DCC_RUN_STOP`

### 11.3 当前任务周期与关键参数

| 参数 | 当前值 |
| --- | --- |
| 主循环周期 | `20ms` |
| PREPARE 持续时间 | `3000ms` |
| 上电额外启动延时 | `3000ms` |
| STRAIGHT 传感器保护窗 | `1000ms` |
| 连续黑线停机阈值 | `3` |

### 11.4 当前运行思路

`DccTask` 的大体逻辑是：

1. 等初始化完成
2. 消费按键信号量
3. 更新模式和运行态
4. 根据当前状态决定是否输出
5. 在 `ON` 态按模式执行控制
6. 遇到保护条件时停机

### 11.5 STRAIGHT 模式

当前 `STRAIGHT` 模式做的是：

1. 用位置环修正左右差异
2. 用左右速度环输出占空比
3. 通过编码器反馈闭环
4. 在保护窗外监测灰度是否连续触发黑线停机条件

### 11.6 TRACK 模式

当前 `TRACK` 模式做的是：

1. 从 `mod_sensor` 获取 12 路状态和权重偏差
2. 根据偏差修正左右轮目标速度
3. 最终输出给 `mod_motor`

### 11.7 STOP 状态的意义

`STOP` 不是普通停机，而是保护态。  
进入后会：

1. 强制停电机
2. 复位 PID
3. 强制把模式回到 `IDLE`

这表示系统明确认为当前状态“不能继续自动执行”。

### 11.8 维护 DCC 时必须注意

1. 新增状态时，仍要保持它是唯一写入者。
2. 所有保护逻辑尽量继续集中在 `DccTask` 内。
3. 不要让别的任务直接写 `s_dcc_mode` 或 `s_dcc_run_state`。
4. 改了模式和运行态逻辑后，要同步更新 OLED 和文档。

## 12. `task_stepper.c` 详解
### 12.1 它的职责

`StepperTask` 负责：

1. 准备双轴步进通道
2. 消费 K230 最新误差帧
3. 根据 DCC 状态决定是否允许控制
4. 下发双轴步进命令
5. 维护软限位、丢帧保护和状态快照

### 12.2 当前任务节拍与关键参数

| 参数 | 当前值 |
| --- | --- |
| 主循环周期 | `10ms` |
| 连续无新帧停机阈值 | `3` 个周期 |
| 每圈脉冲数 | `3200` |
| 位置模式最大速度 | `300 rpm` |
| 速度模式最大速度 | `50 rpm` |
| 误差死区 | `1` |
| X 轴软限位 | `800 pulse` |
| Y 轴软限位 | `400 pulse` |

### 12.3 当前双轴通道分配

1. X 轴使用 `UART5`
2. Y 轴使用 `USART2`
3. 两个驱动地址当前都使用 `1`

### 12.4 当前它依赖什么

1. `K230` 提供视觉误差帧
2. `DCC` 提供当前模式和运行态
3. `mod_stepper` 提供协议发送
4. `VOFA` 可选提供调试输出

### 12.5 当前的保护逻辑

1. 如果没有新视觉帧，允许短时容忍。
2. 连续超过阈值，就停机保护。
3. 如果轴不启用，不发送该轴命令。
4. 如果撞到软限位，不再继续向外推。
5. 如果误差落入死区，发送停机或不继续推动。

### 12.6 前馈设计

当前 `STRAIGHT` 模式下还做了一个有价值的设计：

1. 读取底盘电机累计编码器位置差分
2. 将其作为双轴前馈输入的一部分

这说明步进控制不是完全孤立地只看视觉误差，而是考虑了底盘本身运动。

### 12.7 维护 Stepper 时最重要的点

1. 先确认是通信问题还是控制策略问题。
2. 改驱动参数时，同步检查脉冲数、速度上限、软限位。
3. 改轴定义时，同步检查逻辑轴 ID、方向反相、K230 电机 ID 映射。
4. 改任务状态快照结构时，同步更新 VOFA 和文档。

## 13. `task_test.c` 与 `defaultTask`
### 13.1 `task_test.c`

这是一个测试预留入口。  
它可以用于：

1. 临时联调
2. 快速验证某个模块
3. 做短期实验

但维护时应注意：

1. 临时测试逻辑不要长期堆积。
2. 最终正式行为不要依赖 `TestTask`。

### 13.2 `defaultTask`

这是一个保底空转任务。  
它通常不承载业务逻辑。

## 14. 新增任务时的标准做法
后续新增任务时，推荐按以下顺序处理：

1. 在 `freertos.c` 创建线程对象。
2. 根据需要创建新的信号量或互斥锁。
3. 新建 `task_xxx.h/.c`。
4. 默认在主循环前调用 `task_wait_init_done()`。
5. 如果任务要用模块，优先通过 `InitTask` 或既有装配入口注入绑定结果。
6. 更新本文件和系统总览文档。

## 15. 任务层最容易犯的错误
### 错误 1：在任务里直接操作 GPIO/TIM/UART

这会破坏分层。

### 错误 2：让多个任务同时改一份核心状态

这会让状态机失控。

### 错误 3：新增任务但不接初始化闸门

这样很容易出现抢跑。

### 错误 4：把绑定逻辑塞进业务任务

绑定应尽量集中在 `InitTask`，而不是散在主循环里。

## 16. 任务层维护检查清单
每次改任务层后，建议至少检查：

1. 任务优先级和栈是否还合理。
2. 初始化闸门是否仍然成立。
3. 新增或修改的状态是否只有一个写入者。
4. 任务是否仍只通过模块层访问硬件。
5. 文档、OLED、VOFA、反馈逻辑是否同步更新。

## 17. 这份文档什么时候必须更新
出现下面任一情况，就必须更新本文件：

1. 任务数量变化
2. 任务周期变化
3. 状态机变化
4. 初始化流程变化
5. 同步资源变化
6. 任务职责发生明显调整
