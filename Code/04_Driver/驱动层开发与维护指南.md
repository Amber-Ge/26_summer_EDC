# 驱动层开发与维护指南

## 当前定位

驱动层目录为 `Code/04_Driver`。

它负责提供最小硬件访问能力和稳定接口，位于 HAL 之上、Module 之下。驱动层应尽量保持通用，不携带项目业务语义。

## 当前主干原则

1. 驱动层不描述“这是哪块旧板子上的哪个功能”。
2. 驱动层可以依赖 HAL 句柄类型，但不应在文档层默认假设某个具体外设实例必须存在。
3. 外设实例是否启用，由 CubeMX 和装配层共同决定。
4. 驱动接口语义应尽量长期稳定。

## 关于 UART 的当前口径

`drv_uart` 目前内部仍维护 UART 实例到索引的固定映射表，这是实现层约束，不代表主干工程必须启用这些串口。

更准确的理解应是：

1. 驱动层支持若干 UART 实例登记。
2. 当前工程可按新 PCB 需要启用其中一部分。
3. 未启用的 UART 不属于主干默认绑定要求。

因此，文档中不再把 `USART1`、`USART2`、`USART3`、`USART6` 或任何其他实例写成“当前工程默认必须使用的串口”。

## 当前仍保留价值的驱动

- `drv_gpio`
- `drv_adc`
- `drv_key`
- `drv_pwm`
- `drv_encoder`
- `drv_uart`

这些驱动是否被新板卡实际用到，应由后续硬件方案决定；当前不建议因旧方案退出而删除驱动层实现。

## 后续维护建议

1. 若更换新 PCB，只先调整 `.ioc`、HAL 句柄和装配层绑定。
2. 若只是外设实例变更，尽量不要改驱动接口语义。
3. 只有在新硬件类别明显变化时，才考虑扩展驱动抽象。
