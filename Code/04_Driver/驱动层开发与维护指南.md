# 驱动层开发与维护指南
## 1. 驱动层的定位
驱动层对应目录：

`Code/04_Driver`

它是当前工程里最靠近硬件、但又比 HAL 更稳定的一层。  
它的核心职责是：

1. 提供最小硬件访问能力
2. 做参数校验和状态校验
3. 给模块层提供可组合的底层能力

可将其理解为：

1. HAL 太底层、太散
2. 任务层太高层、太业务
3. 驱动层正好夹在中间，负责建立稳定接口

## 2. 驱动层不应该做什么
驱动层不应该负责：

1. 项目业务状态机
2. 具体设备语义
3. 多任务调度
4. 比如“这是黄灯”“这是底盘左轮”之类的命名

一旦驱动层开始出现明显业务命名，就说明层次开始混乱了。

## 3. 当前驱动总览
| 驱动 | 主要能力 | 典型上层 |
| --- | --- | --- |
| `drv_gpio` | GPIO 读写翻转 | `mod_led/mod_relay/mod_sensor/mod_key/mod_motor` |
| `drv_adc` | ADC 单次与平均采样 | `mod_battery` |
| `drv_key` | 通用按键时序状态机 | `mod_key` |
| `drv_pwm` | PWM 生命周期与占空比控制 | `mod_motor` |
| `drv_encoder` | 编码器启停与增量读取 | `mod_motor` |
| `drv_uart` | UART 阻塞收发、DMA 收发、回调分发 | `mod_vofa/mod_k230/mod_stepper` |

## 4. 当前驱动层统一约束
### 4.1 不写死项目语义

比如：

1. `drv_gpio` 不关心是 LED 还是按键
2. `drv_pwm` 不关心这个 PWM 是给左轮还是右轮
3. `drv_uart` 不关心这是 VOFA 还是 K230

### 4.2 尽量提供稳定的错误语义

当前很多驱动都提供：

1. 参数错误
2. 状态错误
3. HAL 错误

这种区分对模块层做降级处理非常有帮助。

### 4.3 状态化驱动显式生命周期

像 `drv_pwm`、`drv_encoder` 这类驱动，当前都采用清晰生命周期：

1. `ctx_init`
2. `start`
3. `use`
4. `stop`

这是一种值得长期保持的风格。

## 5. `drv_gpio` 详解
### 5.1 它提供什么

`drv_gpio` 只提供 3 件事：

1. `write`
2. `read`
3. `toggle`

### 5.2 为什么它虽然简单但必须存在

因为它统一了：

1. GPIO 逻辑电平抽象
2. 上层访问风格
3. 对 HAL 的隔离

这样模块层以后就不需要反复直接写 HAL API。

## 6. `drv_adc` 详解
### 6.1 它提供什么

`drv_adc` 当前提供：

1. 单次原始值采样
2. 多次平均采样

### 6.2 它为什么只返回原始值

因为“原始值转电压”已经带有设备语义了，应该交给 `mod_battery`。

### 6.3 当前关键参数

当前阻塞轮询超时：

`DRV_ADC_TIMEOUT_MS = 10ms`

## 7. `drv_key` 详解
### 7.1 它的定位

`drv_key` 是一个通用按键驱动，不绑定具体 GPIO。  
它只关心：

1. 某个 key_id 当前有没有按下
2. 消抖
3. 单击
4. 双击
5. 长按

### 7.2 当前默认时序参数

| 参数 | 当前值 |
| --- | --- |
| 最大按键数 | `16` |
| 默认消抖 tick | `2` |
| 默认长按阈值 | `70` |
| 默认双击间隔 | `25` |

### 7.3 为什么它值得保留为独立驱动

因为这种按键时序状态机高度通用，几乎所有项目都可能复用。

## 8. `drv_pwm` 详解
### 8.1 它的定位

`drv_pwm` 封装了单个 PWM 通道的生命周期和占空比控制。  
它当前维护：

1. `htim`
2. `channel`
3. `duty_max`
4. `duty_last`
5. `invert`
6. `started`

### 8.2 当前生命周期

1. `drv_pwm_ctx_init`
2. `drv_pwm_start`
3. `drv_pwm_set_duty`
4. `drv_pwm_stop`

### 8.3 为什么驱动层要支持 `invert`

因为“输出极性差异”属于硬件层面的适配，不应该把这个负担继续往模块层和任务层传。

## 9. `drv_encoder` 详解
### 9.1 它的定位

`drv_encoder` 封装编码器定时器模式的读取。  
当前维护：

1. 定时器句柄
2. 位宽
3. 方向系数
4. 最近增量
5. 启动状态

### 9.2 当前支持的位宽

1. `16 bit`
2. `32 bit`

### 9.3 当前一个非常重要的语义

`drv_encoder_get_delta()` 是“读取后清零”的语义。  
这点不能轻易改。

因为上层 `mod_motor` 正是基于这个语义按周期统计速度反馈。

### 9.4 当前为什么支持反向

因为不同电机安装和接线会造成方向定义差异。  
把方向修正放在驱动层，可以保持上层速度符号语义统一。

## 10. `drv_uart` 详解
### 10.1 它是当前最重要的基础驱动之一

`drv_uart` 负责：

1. 阻塞发送
2. 阻塞读取
3. DMA 发送
4. ReceiveToIdle DMA 接收
5. RX 事件分发
6. 回调注册
7. user_ctx 透传
8. UART 实例到固定端口索引的映射

### 10.2 当前支持的端口数量

当前支持：

`6` 个 UART 端口映射

用于统一管理：

1. `USART1`
2. `USART2`
3. `USART3`
4. `UART4`
5. `UART5`
6. `USART6`

### 10.3 当前为什么它这么重要

因为：

1. `VOFA`
2. `K230`
3. `Stepper`

都依赖它。  
如果 `drv_uart` 的接口混乱，整个协议层都会变得难维护。

### 10.4 当前比较好的设计点

1. 支持 `drv_uart_status_t`
2. 支持 `drv_uart_rx_event_t`
3. 支持 `register_rx_callback(..., user_ctx)`
4. 支持关闭 RX DMA 半传输中断

这说明它不是只够用，而是已经在往“统一协议底座”的方向设计。

### 10.5 当前维护它时最该注意什么

1. 不要把协议解析写进驱动层回调。
2. UART 端口映射变了，要同步看所有协议模块。
3. 状态码语义变了，模块层的降级逻辑要同步检查。

## 11. 驱动层与模块层的接口关系
模块层默认应该：

1. 消费驱动接口
2. 根据驱动状态码决定如何处理
3. 把驱动能力封装成设备语义

任务层则不应直接调用驱动层。  
如果发现任务层开始频繁直接调用驱动层，通常说明模块抽象仍然不够。

## 12. 新增驱动时的建议
以后新增驱动时，建议先判断：

1. 这是原子硬件能力，还是设备语义？
2. 如果只是原子能力，才放 `Driver`
3. 如果已经带有设备含义，优先考虑放 `Module`

### 推荐风格

如果是状态化驱动，建议采用：

1. `ctx`
2. `status`
3. `init/start/stop/use`

如果是简单原子驱动，建议接口保持简单直接。

## 13. 驱动层最容易犯的错误
### 错误 1：驱动层出现业务命名

这会导致层次污染。

### 错误 2：驱动层隐藏太多状态

会让上层很难判断是否 ready。

### 错误 3：驱动层接口语义不稳定

尤其像 `encoder get_delta` 这种基础接口，一旦语义改动，影响面很大。

### 错误 4：把 HAL 细节直接泄漏到所有上层

驱动层存在的意义，就是减少这种泄漏。

## 14. 驱动层维护检查清单
每次改驱动层后，建议检查：

1. 参数校验是否完整
2. 状态校验是否完整
3. 旧接口语义是否被破坏
4. 依赖它的模块是否需要同步调整
5. 文档是否同步更新

## 15. 这份文档什么时候必须更新
出现下面任一情况，就必须更新本文件：

1. 新增驱动
2. 删除驱动
3. 驱动接口变化
4. 驱动状态语义变化
5. 驱动生命周期变化
