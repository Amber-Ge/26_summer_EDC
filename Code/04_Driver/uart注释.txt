. 总体设计思想 (Design Philosophy)
本项目的 PC 通信系统（基于 VOFA+ 上位机）采用 “四层解耦” 架构。其核心目标是：中断负责翻译，任务负责业务，硬件负责搬运。

硬件层 (Hardware): STM32ZGT6 UART + DMA。

驱动层 (Driver): drv_uart（负责中断分发与 DMA 资源管理）。

模组层 (Module): mod_vofa（负责字符串匹配、协议解析与信号量释放）。

任务层 (Task): task_pc（负责逻辑状态切换、数据上报决策）。

2. 核心全流程解析 (Full Workflow)
A. 接收流程 (RX: 指令下发)
这是最精妙的部分，解决了“不定长字符串”解析的实时性问题：

静默搬运：上位机发送指令（如 start\n）。DMA 在后台自动将数据搬入 s_vofa_rx_buf，此过程 0% CPU 占用。

空闲检测：一帧数据发完后，串口总线进入空闲状态，触发 IDLE (空闲中断)。

驱动路由：drv_uart.c 的 IDLE 回调被触发，它根据串口号找到已登记的 _vofa_rx_callback_handler。

模组匹配：

使用 strstr 扫描缓冲区，匹配 s_cmd_table 中的关键词。

关键动作：匹配成功后，必须先 HAL_UART_DMAStop，再 memset 清空缓冲区，最后重启 DMA。这是为了彻底消除旧指令残留和重置 DMA 偏移地址。

信号同步：模组通过 osSemaphoreRelease 释放信号量，通知 task_pc。

逻辑触发：task_pc 醒来，拿到指令 ID，执行动作（如开启发送开关）。

B. 发送流程 (TX: 波形/数据上报)
非阻塞调用：任务层调用 mod_vofa_send_float。

防撞检查：模组先调用 drv_uart_is_tx_free 确认硬件传送带（DMA TX）是否空闲。

格式化：将数值转换为 FireWater 协议格式（Tag:v1,v2\n）。

异步发送：驱动层启动 DMA 发送后立刻返回，任务继续跑，硬件在后台慢慢发。

3. 开发者指南：如何添加新指令 (Developer Guide)
如果你需要增加一个功能（例如：发送 reset 让小车复位）：

定义 ID：在 mod_vofa.h 的 vofa_cmd_id_t 枚举中添加 VOFA_CMD_RESET。

注册关键词：在 mod_vofa.c 的 s_cmd_table 中添加 {"reset", VOFA_CMD_RESET}。

编写动作：在 task_pc.c 的指令处理区增加分支：

C
else if (cmd == VOFA_CMD_RESET) {
    // 在这里写你的复位逻辑代码
}
4. 关键注意事项 (The Pitfalls - 必读！)
1. 信号量注册：mod_vofa_init 必须在任务循环开始前调用，并传入对应的信号量。如果传 NULL，指令解析仍会进行，但无法即时唤醒任务。

2. 缓冲区对齐：不要尝试手动修改接收缓冲区。所有的清空和重置逻辑已在 mod_vofa 内部闭环，手动修改会导致 DMA 地址偏移，出现“发几次才成功一次”的灵异现象。

3. 发送频率限制：由于 115200 波特率的带宽限制，建议 TASK_PC_PERIOD_MS 设为 20ms (50Hz)。如果发送路数超过 10 路，请降频至 50ms。

4. 硬件配置：CubeMX 中串口的 RX DMA 必须配置为 Circular (循环模式)，否则 IDLE 接收只能成功一次。

5. 字符串唯一性：指令关键词不要互相包含（例如不要同时定义 ok 和 okay），否则 strstr 会发生误判。

5. 调试建议 (Debug Tips)
验证接收：发指令后，观察 VOFA 文本窗口是否有 StartAck 或 StopAck 回传。如果没有，说明 RX 链路不通（检查 DMA 或中断优先级）。

验证发送：观察波形是否跳变。如果波形不动，检查 drv_uart_is_tx_free 是否一直返回 false（可能是串口线松了或硬件错误）。
