# 模块层开发与维护指南

## 当前定位

模块层位于 `Task` 与 `Driver` 之间，负责把底层驱动能力组织成设备语义和可绑定的上下文。

当前工程中，模块层仍然是长期可复用资产层。后续更换 PCB 时，应优先通过装配层重新绑定模块，而不是直接改写模块内部逻辑。

## 当前主干原则

1. 模块层不直接承担 FreeRTOS 任务调度。
2. 模块层不直接写死某个旧板卡的外设实例。
3. 模块层通过 `ctx + bind` 方式接收外设句柄和配置。
4. 新板卡的实例选择由 `InitTask` 或后续装配层决定。

## 与装配层的关系

当前工程已经收缩为“主干只保留 InitTask 框架”的状态。

因此应这样理解：

1. `InitTask` 负责决定“当前平台要不要启用某个模块”。
2. `InitTask` 负责决定“某个模块绑定到哪个 UART / TIM / ADC / GPIO”。
3. 模块层只负责消费这些绑定，不再默认假设旧工程中的 `USART3`、`UART4`、`TIM4` 等实例一定存在。

## 仍然保留价值的模块

- `mod_led`
- `mod_relay`
- `mod_sensor`
- `mod_key`
- `mod_motor`
- `mod_battery`
- `mod_oled`
- `mod_vofa`
- `mod_k230`
- `mod_stepper`
- `mod_uart_guard`

这些模块是否在当前版本启用，由装配层决定；不应因为暂时未绑定新板卡就删除。

## 当前需要避免的旧口径

后续维护时，不要再把以下表述写回主干文档：

1. “InitTask 默认绑定 USART3 给 VOFA”
2. “K230 固定接 UART4”
3. “Stepper 固定走 USART2/UART5”
4. “某个旧传感器表一定是默认主配置”

这些都属于历史项目实例，而不是当前主干架构要求。
