# 模块层开发与维护指南
## 1. 模块层的定位
模块层位于：

`Task` 和 `Driver` 之间

它的职责不是做任务调度，也不是做最底层硬件访问，而是：

1. 把原子驱动能力组合成设备语义
2. 把硬件映射和运行状态收敛到统一上下文
3. 对上层提供稳定、可读、可复用的接口

例如：

1. `drv_gpio` 只知道“写高写低”
2. `mod_led` 才知道“这是红灯还是绿灯、是不是低电平有效”

这就是模块层存在的意义。

## 2. 模块层的统一设计思想
### 2.1 `ctx + bind` 模式

当前工程里绝大多数模块都采用了统一风格：

1. `get_default_ctx`
2. `ctx_init`
3. `bind`
4. `unbind`
5. `is_bound`
6. 运行期 API

这种模式带来的好处是：

1. 生命周期清楚
2. 绑定关系清楚
3. 状态集中
4. 后续移植和复用更方便

### 2.2 模块层应该做什么

模块层应该负责：

1. 硬件映射表的消费
2. 运行期上下文管理
3. 语义化接口封装
4. 协议帧编解码
5. 错误降级或就绪性判断

### 2.3 模块层不应该做什么

模块层不应该负责：

1. FreeRTOS 任务调度
2. 整机状态机
3. 跨任务同步策略
4. 答案式业务逻辑编排

## 3. 当前模块总览
| 模块 | 主要作用 | 主要下层依赖 |
| --- | --- | --- |
| `mod_led` | 语义化 LED 控制 | `drv_gpio` |
| `mod_relay` | 语义化继电器控制 | `drv_gpio` |
| `mod_sensor` | 12 路灰度采样与权重计算 | `drv_gpio` |
| `mod_key` | 按键映射和事件语义输出 | `drv_key + drv_gpio` |
| `mod_motor` | 双直流电机执行器抽象 | `drv_gpio + drv_pwm + drv_encoder` |
| `mod_battery` | 电池电压采样与换算 | `drv_adc` |
| `mod_oled` | OLED 显示语义封装 | OLED/I2C 组件 |
| `mod_vofa` | VOFA 文本通信模块 | `drv_uart + mod_uart_guard` |
| `mod_k230` | K230 自定义协议接收解析 | `drv_uart + mod_uart_guard` |
| `mod_stepper` | 步进驱动协议发送模块 | `drv_uart + mod_uart_guard` |
| `mod_uart_guard` | UART 资源归属仲裁 | `drv_uart` |

## 4. 简单语义模块
### 4.1 `mod_led`

它的作用不是简单“写 GPIO”，而是：

1. 把 LED 通道变成逻辑设备
2. 支持 `active_level`
3. 让上层不必关心高低电平有效差异

当前工程里它被用于：

1. 红灯
2. 绿灯
3. 黄灯

### 4.2 `mod_relay`

它和 `mod_led` 风格一致，区别只是设备语义不同。  
当前用于：

1. 激光继电器
2. 蜂鸣器继电器

### 4.3 为什么这类简单模块仍然值得保留

因为如果没有它们：

1. 任务层里会到处写 GPIO
2. 高低电平有效逻辑会分散
3. 换板时改动范围会变大

## 5. `mod_sensor` 详解
### 5.1 它做了什么

`mod_sensor` 负责：

1. 维护 12 路灰度输入映射
2. 逐路读取当前是否检测到黑线
3. 计算归一化权重偏差

### 5.2 它的输出不是“直接控制命令”

这一点非常关键。  
`mod_sensor` 输出的是：

1. `states[]`
2. `weight`

它不负责：

1. 左转还是右转
2. 电机占空比是多少

这些事情应该留给 `DccTask`。

### 5.3 当前绑定方式

当前在 `task_init.c` 中，把 12 路引脚和权重表注入 `mod_sensor`。  
因此后续如果调整传感器顺序，通常只需修改绑定和权重，而不必改模块算法。

## 6. `mod_key` 详解
### 6.1 它在任务层和驱动层之间做了什么

底层 `drv_key` 只输出：

1. 某个 key_id
2. click / double click / long press

而 `mod_key` 负责把这些通用事件映射成当前工程需要的模块事件：

1. `MOD_KEY_EVENT_1_CLICK`
2. `MOD_KEY_EVENT_2_CLICK`
3. `MOD_KEY_EVENT_3_DOUBLE_CLICK`
4. 等等

### 6.2 为什么不直接在 `drv_key` 里写业务事件

因为 `drv_key` 应当是通用的。  
如果把业务事件写进驱动层，复用价值会立刻下降。

## 7. `mod_battery` 详解
### 7.1 它解决什么问题

`drv_adc` 只输出原始采样值。  
`mod_battery` 则负责把它变成“可读的电压值”。

当前它会处理：

1. ADC 句柄
2. 参考电压
3. ADC 分辨率
4. 分压恢复系数
5. 平均采样次数

### 7.2 当前默认参数

| 参数 | 当前值 |
| --- | --- |
| 参考电压 | `3.3V` |
| ADC 满量程 | `4095` |
| 分压恢复系数 | `10` |
| 默认平均采样次数 | `10` |

## 8. `mod_motor` 详解
### 8.1 它是当前最重要的设备模块之一

`mod_motor` 把下面 3 类能力组合到一起：

1. 方向 GPIO
2. PWM 输出
3. 编码器反馈

从而对上层提供“一个电机”的抽象。

### 8.2 当前支持的逻辑通道

1. `MOD_MOTOR_LEFT`
2. `MOD_MOTOR_RIGHT`

### 8.3 当前支持的运行模式

1. `MOTOR_MODE_DRIVE`
2. `MOTOR_MODE_BRAKE`
3. `MOTOR_MODE_COAST`

### 8.4 它内部维护什么状态

每路电机都会维护：

1. 当前模式
2. 最近方向符号
3. 过零挂起标志
4. 挂起占空比
5. 当前速度反馈
6. 累计位置
7. 硬件是否 ready

### 8.5 为什么这个模块重要

因为它把当前工程里最复杂的执行器抽象集中在了一处。  
如果没有它，`DccTask` 就会直接面对：

1. GPIO 方向控制
2. PWM 占空比
3. 编码器增量
4. 方向修正
5. 过零保护

任务层会迅速变得非常难维护。

### 8.6 维护它时要注意什么

1. 改方向逻辑时，要同时看电机方向和编码器方向。
2. 改占空比范围时，要和 PWM 的 `duty_max` 对齐。
3. 新增保护逻辑时，优先收敛在模块内部，不要散到任务层。

## 9. `mod_oled` 详解
### 9.1 它的定位

`mod_oled` 负责的是显示设备接口，不是业务页面设计。  
任务层决定显示什么，模块层提供显示能力。

### 9.2 当前工程里的使用方式

当前 `OledTask` 主要调用 OLED 组件来显示：

1. 文本
2. 数值
3. 状态字符串

后续如果需要扩展复杂页面，也建议继续保持：

1. 模块层负责显示能力
2. 任务层负责页面内容编排

## 10. UART 协议模块组总览
当前工程里最有代表性的模块组，是基于 UART 的协议模块组：

1. `mod_vofa`
2. `mod_k230`
3. `mod_stepper`
4. `mod_uart_guard`

它们一起体现了当前工程在“串口协议工程化管理”上的思路。

## 11. `mod_uart_guard` 详解
### 11.1 它解决什么问题

如果多个协议模块都可能绑定 UART，就会遇到：

1. 谁拥有这个 UART
2. 重复绑定怎么办
3. 同 owner 多实例怎么办
4. 解绑时是否会误释放其他模块资源

`mod_uart_guard` 就是专门解决这类问题的。

### 11.2 当前设计要点

1. 维护 UART owner
2. 维护 claim depth
3. 支持 claimant 指针
4. 只允许匹配的 owner + claimant 释放

### 11.3 为什么它值得保留

这块设计非常适合作为后续串口协议模块的标准基建。

## 12. `mod_vofa` 详解
### 12.1 它做了什么

`mod_vofa` 负责：

1. 发送调试数据
2. 接收简单命令
3. 维护发送缓冲和接收缓冲

### 12.2 当前主要接口

当前可以发送：

1. 浮点数组
2. 有符号整数数组
3. 无符号整数数组
4. 字符串

### 12.3 当前命令支持

当前能够识别的命令主要是：

1. `start`
2. `stop`

### 12.4 当前使用方式

1. `InitTask` 绑定 `USART3`
2. 某些任务按需向 VOFA 发送调试数据

## 13. `mod_k230` 详解
### 13.1 它的定位

`mod_k230` 是当前工程里针对 `K230` 自定义协议的接收解析模块。  
它不负责控制步进，也不负责决策，只负责把串口字节流变成结构化误差帧。

### 13.2 当前协议特征

当前协议为固定 12 字节格式，大致为：

1. 帧头：`AA AA`
2. 固定长度字段：`06`
3. 两组逻辑电机 ID 和误差值
4. 校验
5. 帧尾：`55 55`

### 13.3 当前校验方式

当前使用：

`XOR`

校验区间是协议定义中的数据字段部分。

### 13.4 当前模块内部做了什么

1. DMA 接收
2. 软件环形缓冲
3. 固定长度帧解析
4. 校验验证
5. 输出最新有效帧

### 13.5 为什么它做得比“直接在回调里解析”更好

因为它把：

1. DMA 接收
2. 缓冲
3. 解析状态机
4. 校验

全部封装在一个模块里，任务层只拿结果。

## 14. `mod_stepper` 详解
### 14.1 它的定位

`mod_stepper` 是步进驱动协议发送模块。  
它是 TX Only 模块，负责：

1. 组帧
2. 发送
3. 维护发送活动状态
4. 处理发送超时恢复

### 14.2 它不负责什么

它不负责：

1. 视觉误差闭环
2. 多轴协调逻辑
3. 软限位策略
4. DCC 状态关联

这些应该留在 `StepperTask`。

### 14.3 当前命令类型

根据当前实现，主要支持：

1. 使能命令
2. 速度命令
3. 位置命令
4. 停止命令

对应协议命令字包括：

1. `F3`
2. `F6`
3. `FD`
4. `FE`

帧尾使用：

`0x6B`

### 14.4 当前模块设计亮点

1. 发送前统一走 `_send_frame`
2. 支持互斥锁保护
3. 不依赖全局 TX 完成回调
4. 通过轮询硬件空闲状态推进发送状态

## 15. 模块层和任务层如何分工
一个简单判断标准是：

1. 如果某段逻辑仍在描述“这个设备是什么、怎么收发、怎么换算”，通常仍属于 `Module`
2. 如果某段逻辑已经在描述“什么时候做、按什么状态切换、多个模块怎么配合”，通常应该归入 `Task`

例如：

1. “步进位置命令怎么组帧”属于 `Module`
2. “什么情况下给步进下位置命令”属于 `Task`

## 16. 新增模块的推荐模板
以后新增模块时，建议至少保留下面这些组成：

1. `mod_xxx_bind_t`
2. `mod_xxx_ctx_t`
3. `mod_xxx_get_default_ctx`
4. `mod_xxx_ctx_init`
5. `mod_xxx_bind`
6. `mod_xxx_unbind`
7. `mod_xxx_is_bound`
8. 运行期接口

如果是协议类模块，还建议同时考虑：

1. DMA 缓冲
2. 软件环形缓冲
3. user_ctx 回调
4. UART 资源仲裁
5. 发送互斥

## 17. 模块层最容易犯的错误
### 错误 1：把任务逻辑塞进模块层

例如在模块里写状态机和任务时序，这会让层次变乱。

### 错误 2：模块层直接依赖具体项目业务

这样会降低复用价值。

### 错误 3：模块层接口设计不统一

如果有的模块用 `ctx`，有的模块不用，上下文风格就会越来越乱。

### 错误 4：绑定和运行状态混在一起

绑定配置应尽量是静态输入，运行状态应放在上下文里单独维护。

## 18. 模块层维护检查清单
每次修改模块层后，建议检查：

1. 生命周期是否仍然清楚
2. bind 和运行态是否仍然解耦
3. 上层任务是否仍通过模块接口调用
4. 修改是否破坏了复用性
5. 文档是否同步更新

## 19. 这份文档什么时候必须更新
出现下面任一情况，就必须更新：

1. 新增模块
2. 删除模块
3. 模块生命周期变化
4. 模块接口变化
5. 协议格式变化
6. 绑定方式变化
