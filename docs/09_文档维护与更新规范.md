# 文档维护与更新规范

## 1. 文档在当前工程中的地位

当前工程的目标不是“代码能跑就结束”，而是：

1. 让后续接手者能尽快理解层次与职责
2. 让复用资产长期沉淀
3. 让归档与主干边界清晰可维护

因此，文档本身也是工程的一部分。

## 3. 什么时候必须同步更新文档

### 3.1 改了 `.ioc` 或硬件绑定

至少检查：

1. `README.md`
2. `docs/04_...`
3. `docs/05_...`
4. 如涉及启动链变化，还要检查 `docs/06_...`

### 3.2 改了 `freertos.c`

至少检查：

1. `README.md`
2. `docs/06_系统启动流程与运行逻辑总览.md`
3. `Code/01_Task/任务层开发与维护指南.md`

### 3.3 改了 `task_init.c`

至少检查：

1. `docs/06_系统启动流程与运行逻辑总览.md`
2. `Code/01_Task/任务层开发与维护指南.md`
3. 如涉及绑定表或映射，还要检查 `docs/05_...`

### 3.4 改了目录结构、归档规则或主干边界

至少检查：

1. `README.md`
2. `docs/10_目录分层与归档说明.md`
3. 相关层级手册

## 4. 归档规则

当某个任务、模块、驱动或说明文档满足以下条件时，优先归档到 `99_Archive`，不要直接删除：

1. 当前默认不参与编译或启动
2. 未来仍可能复用
3. 保留在主干会增加理解成本或边界混乱

归档后必须确认：

1. `CMakeLists.txt` 没有把 `99_Archive` 收进默认构建
2. 主干 include 路径没有被归档文件反向依赖
3. README 和相关手册已注明“归档而非删除”

## 5. 文档写法要求

本工程后续文档维护应遵守：

1. 以“在现有体系上的保守演进”为表述口径
2. 不把现有 `Task -> Module -> Driver -> HAL` 描述成错误设计
3. 不为了追求术语更标准而否定当前 `InitTask` 装配入口
4. 对“未来可优化”与“本次未实施”做明确区分
