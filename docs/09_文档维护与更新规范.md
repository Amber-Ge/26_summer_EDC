# 文档维护与更新规范
## 1. 为什么这个工程必须重视文档
这个工程的目标不只是“代码能跑”，而是：

1. 让后续读者能够复现
2. 让后续维护者能够维护
3. 让后续开发者能够继续扩展
4. 让项目维护者在数月后重新阅读时仍能快速恢复理解

如果没有文档，这套分层结构的价值会快速下降。  
因为后续读者虽然看到目录很多，但并不知道：

1. 哪个文件是入口
2. 哪个层负责什么
3. 改一个功能应该先动哪一层
4. 改完之后哪些地方必须联动更新

因此在这个仓库里，文档不是附属品，而是工程的一部分。

## 3. 哪些代码变化必须同步改文档
### 3.1 改了 `.ioc`

必须至少检查并更新：

1. `README.md`
2. `docs/04_CubeMX引脚修改与工程维护指南.md`
3. `docs/05_硬件连接与外设映射总表.md`

如果涉及系统流程，也要同步：

1. `docs/06_系统启动流程与运行逻辑总览.md`

### 3.2 改了 `freertos.c`

必须至少检查并更新：

## 4. 文档更新的最低要求
当项目代码发生修改时，文档更新至少要做到下面 4 件事：

1. 把“变了什么”写出来
2. 把“为什么变”写出来
3. 把“影响到哪一层”写出来
4. 把“后续维护者应该注意什么”写出来

只写一句“更新文档”是没有意义的。  
必须让后来者知道变化的上下文。

## 6. 文档命名规则
当前仓库采用中文文件名，是为了：

1. 新手更直观
2. 阅读成本更低
3. 顶层索引更清晰

命名建议继续遵守：

1. 总览类文档放在 `docs/`
2. 使用 `数字_中文主题.md`
3. 分层文档放在各层目录下
4. 文件名直接体现用途，避免抽象词过多

## 7. 文档内容写作规则
### 7.1 先说给谁看

每份文档开头建议先写：

1. 这份文档给谁看
2. 解决什么问题
3. 不解决什么问题

### 7.2 先讲入口，再讲细节

一份文档应该尽量让读者先快速定位，再深入阅读。  
不建议在开头铺陈过长的背景说明，却不告诉读者“接下来应该看哪里”。

### 7.5 叙述视角保持客观

默认应采用工程说明书视角书写，建议使用：

1. `本工程`
2. `本项目`
3. `读者`
4. `复现者`
5. `维护者`
6. `目标项目`

不建议长期使用下面这类作者本人视角表达：

1. 第二人称单数表达
2. “作者本人拥有该工程”的表达
3. “作者本人亲自操作”的表达
4. “以后由其他人接手”的口语化表达

除非某一段明确是在记录作者操作日志或提交说明，否则文档应保持第三方、客观、可公开阅读的表述方式。

### 7.3 说明必须和代码一致

不能写“理论上应该怎样”，却和当前仓库实现不一样。  
当前仓库文档默认优先描述：

1. 当前代码真的做了什么
2. 当前参数真的是多少
3. 当前绑定真的是什么

### 7.4 变化时要写绝对信息

尤其是：

1. 外设实例
2. 引脚
3. 参数值
4. 周期
5. 阈值

不要只写：

1. “稍后更新”
2. “有一些变化”
3. “大概如此”

## 8. 每次提交前的文档检查表
在准备提交代码前，至少检查下面这些问题：

1. `README.md` 索引是否仍然有效
2. 新增的 Markdown 是否已经被入口索引到
3. 文件名是否清晰
4. 文档中提到的参数、任务名、外设名是否和代码一致
5. 改了引脚后，硬件总表是否同步
6. 改了任务逻辑后，系统总览是否同步
7. 改了模块接口后，分层手册是否同步

## 9. 推荐的文档维护流程
建议以后每次做结构性修改时，按这个顺序维护文档：

1. 先改代码
2. 再确认代码最终实现
3. 然后更新对应分层手册
4. 再更新 `docs/` 下的总览类文档
5. 最后回到 `README.md` 检查入口索引

这样做的好处是：

1. 不容易漏
2. 不容易写出和代码不一致的内容
3. 顶层入口始终保持可信

## 10. 文档失效的典型信号
如果出现下面这些情况，说明文档已经开始失效：

1. 读者按文档操作却跑不起来
2. 文档里的引脚和 `.ioc` 对不上
3. 文档里的任务数和 `freertos.c` 对不上
4. 文档里的接口名和头文件对不上
5. 文档描述的是“旧结构”，代码已经改成“新结构”

一旦出现这些信号，不要继续在错误文档上叠加补丁，而应该整体重读对应部分并重写。

## 11. 推荐的维护原则总结
以后维护这个仓库时，建议一直遵守下面 5 条：

1. 文档跟代码一起维护，不要积压。
2. 结构变化先更新分层手册，再更新总览。
3. 引脚变化必须同步更新硬件总表。
4. 入口文档必须始终可跳转、可导航。
5. 对外说明一律优先写“当前已经验证过的能力”，不要文档先于实现夸大。

## 12. 这份规范本身什么时候应该更新
如果未来调整了：

1. 文档目录结构
2. 文件命名规则
3. 更新流程
4. 入口索引方式
5. 分层手册组织方式

就要优先更新这份规范，因为它是整个文档系统的“规则说明”。
