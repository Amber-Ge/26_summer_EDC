# 烧录、调试、联调指南

## 1. 这份文档的目标

本文件解决 3 件事：

1. 怎么把程序烧进去
2. 怎么开始调试
3. 怎么按正确顺序联调各个模块

对于新手来说，这 3 件事比“直接改代码”更重要。

## 2. 烧录前准备

### 2.1 必须准备的设备

1. `ST-LINK`
2. 主控板
3. 供电
4. 电脑

### 2.2 推荐准备的设备

1. USB 转串口模块
2. 串口助手
3. `VOFA+`
4. 万用表

## 3. 推荐的上电和联调原则

### 3.1 不要一次把所有执行器都接满

最常见的错误就是：

1. 灰度
2. 直流电机
3. 步进
4. K230
5. OLED
6. 串口

全部接好以后一起通电，然后出了问题完全不知道是哪一环。

### 3.2 正确顺序

建议严格按照下面顺序来：

1. 先烧录下载
2. 再确认系统能启动
3. 再确认 OLED、按键、LED、蜂鸣器这些低风险模块
4. 再确认普通串口和 VOFA
5. 再确认灰度传感器
6. 再确认底盘电机方向、编码器、PWM
7. 最后确认 K230 和双轴步进

## 4. 用 ST-LINK 烧录

### 4.1 最基础的 SWD 连接

至少要保证：

1. `SWDIO`
2. `SWCLK`
3. `GND`
4. 电源链路

### 4.2 下载成功的最低标准

成功下载后，至少应确认：

1. 板子没有复位死循环
2. 任务调度开始运行
3. OLED 或基础外设有反应

## 5. 在 CLion 中调试

### 5.1 本工程已有的思路

仓库中的 `.idea/debugServers/ST_LINK.xml` 已经说明这个工程原本就是按 `ST-LINK GDB Server` 方案使用的。

这意味着在新电脑上通常只需要做两件事：

1. 确认 `STM32CubeCLT` 或 `ST-LINK GDB Server` 的实际路径
2. 在 `CLion` 中把调试服务器路径改成当前环境中的实际安装位置

### 5.2 为什么推荐在 Debug 下调试

因为：

1. 符号更完整
2. 单步更容易
3. 新手更容易看清楚任务启动流程

## 6. 首次调试建议观察的函数

### 6.1 程序启动链路

建议先按顺序看：

1. `main`
2. `SystemClock_Config`
3. `MX_GPIO_Init`
4. `MX_DMA_Init`
5. `MX_USART3_UART_Init`
6. `MX_ADC1_Init`
7. `MX_TIM4_Init`
8. `MX_TIM1_Init`
9. `MX_TIM3_Init`
10. `MX_TIM5_Init`
11. `MX_USART1_UART_Init`
12. `MX_USART2_UART_Init`
13. `MX_USART6_UART_Init`
14. `MX_I2C2_Init`
15. `osKernelInitialize`
16. `MX_FREERTOS_Init`
17. `osKernelStart`

### 6.2 任务初始化链路

进入 RTOS 后建议重点看：

1. `StartInitTask`
2. `task_init_bind_map`
3. `task_init_modules`
4. `task_init_vofa_bind`
5. `task_init_k230_bind`
6. `task_init_stepper_bind`
7. `task_wait_init_done`

### 6.3 运行态链路

再往后建议看：

1. `StartDccTask`
2. `StartStepperTask`
3. `StartKeyTask`
4. `StartGpioTask`
5. `StartOledTask`

## 7. 各功能的联调顺序

### 7.1 先调基础输出

建议先确认这些：

1. 红绿黄 LED
2. 蜂鸣器继电器
3. 激光继电器
4. OLED

这样做的原因是：

1. 风险低
2. 肉眼可见
3. 出错容易定位

### 7.2 再调输入

建议先调：

1. 3 个按键
2. 12 路灰度传感器

先把输入链路调通，后面的状态机才容易看明白。

### 7.3 再调底盘

底盘联调顺序建议是：

1. 单独确认左右电机转向
2. 单独确认左右编码器方向
3. 再看 `DccTask` 的 OFF/PREPARE/ON/STOP
4. 再看 `IDLE/STRAIGHT/TRACK`

### 7.4 最后调 K230 和 Stepper

这是因为它最复杂，涉及：

1. K230 自定义协议
2. `USART2` 接收
3. Stepper 双轴串口发送
4. `DccTask` 运行状态门控
5. 软限位
6. 丢帧保护

## 8. 串口联调建议

### 8.1 VOFA

VOFA 当前绑定在：

1. `USART1`
2. `PA9` TX
3. `PA10` RX
4. 波特率 `115200`

典型用途：

1. 查看 `DccTask` 状态
2. 查看 `StepperTask` 误差数据
3. 做 PID 联调

### 8.2 K230

K230 当前绑定在：

1. `USART2`
2. `PD5` TX
3. `PD6` RX
4. 波特率 `115200`

文档中应明确说明：

1. 这里不是现成公开协议
2. 这里使用的是项目自定义的固定长度协议
3. 真正的帧格式要看 `mod_k230`

### 8.3 步进驱动串口

X/Y 两轴分别是：

1. X 轴：`USART3`
2. Y 轴：`USART6`

注意事项：

1. 必须共地
2. 不要把 TX/RX 接反
3. 首次联调要限制机械风险

## 9. 调试时最常见的错误现象

### 9.1 下载成功，但程序看起来“没反应”

先检查：

1. 是否真的进入了 `osKernelStart`
2. `InitTask` 是否执行完成
3. 业务任务是否卡在 `task_wait_init_done`
4. OLED 是否正确绑定了 `I2C2`

### 9.2 串口收发完全异常

先检查：

1. 共地是否正确
2. 波特率是否一致
3. TX/RX 是否接反
4. 串口是否被其他模块占用
5. 是否走了 `mod_uart_guard`

### 9.3 电机方向不对

不要立刻改原理图。

应先判断是哪一层的问题：

1. 是引脚接线真的错了
2. 还是绑定表故意做了软件修正
3. 还是模块层做了方向反转

### 9.4 编码器方向不对

当前工程里右轮编码器方向就是通过软件修正的，所以调试时必须先看：

1. `task_init.c` 的电机绑定
2. `mod_motor` 的编码器绑定参数
3. `drv_encoder` 的 `invert`

## 10. 推荐的首次验证清单

可用下面这个清单判断当前联调进度：

| 项目 | 通过标准 |
| --- | --- |
| 构建 | 能生成 `.elf` |
| 烧录 | 能正常下载到板子 |
| 启动 | 系统启动后不死机 |
| OLED | 有稳定显示 |
| 按键 | 按下后有灯光/蜂鸣反馈 |
| LED | 红绿黄状态符合预期 |
| 蜂鸣器/激光 | 继电器动作正常 |
| 电池采样 | 显示值合理 |
| 灰度输入 | 能读到状态变化 |
| 底盘 | 左右轮方向、编码器方向正确 |
| VOFA | 能收到曲线数据 |
| K230 | 能收到有效误差帧 |
| Stepper | 双轴能按控制逻辑响应 |

## 12. 一句话总结

联调的核心不是“把所有东西都点亮”，而是按风险从低到高、按依赖从底到上、按数据流从输入到输出，逐段验证这套工程到底是不是按文档描述在运行。
