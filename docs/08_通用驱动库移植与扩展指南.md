# 通用驱动库移植与扩展指南
## 1. 这份文档的目标
这份文档不是说明“如何第一次把当前工程跑起来”，而是说明：

1. 如何把这套工程里的可复用部分拆出来。
2. 如何把它移植到新的 STM32 工程里。
3. 如何在保持分层结构的前提下扩展新模块和新任务。

如果目标只是复现当前仓库，请先看：

1. [01_新手从零开始复现工程.md](01_新手从零开始复现工程.md)
2. [04_CubeMX引脚修改与工程维护指南.md](04_CubeMX引脚修改与工程维护指南.md)

## 2. 哪些部分最适合复用
当前仓库中，复用价值最高的通常不是整个项目整体，而是下面三层：

1. `Code/04_Driver`
2. `Code/02_Module`
3. `Code/03_Common`

其中：

1. `Common`
   - 最容易移植
   - 硬件依赖最少
2. `Driver`
   - 和 STM32 外设强相关
   - 但复用价值很高
3. `Module`
   - 依赖上面的 Driver
   - 适合连同 Driver 一起移植

最不适合原样照搬的是：

1. `Task`
2. `Core`
3. 当前工程的具体 `.ioc`

因为这三部分最强依赖“当前工程的硬件平台、当前业务和当前任务设计”。

## 3. 建议把这套工程拆成两类内容理解
### 3.1 通用能力

这些更像“库”：

1. GPIO 驱动
2. PWM 驱动
3. Encoder 驱动
4. ADC 驱动
5. UART 驱动
6. 按键通用驱动
7. LED/Relay/Motor/Sensor/Battery/VOFA/K230/Stepper 模块
8. PID 和通用工具函数

### 3.2 项目定制能力

这些更像“项目业务”：

1. DCC 状态机
2. 当前 3 个按键的业务映射
3. 当前 12 路灰度权重
4. 当前步进双轴控制策略
5. 当前 OLED 页面内容
6. 当前毕设/训练场景下的运行流程

理解这点后，移植时就不应直接整体复制整个项目，而应先判断需要拆分和保留哪些层。

## 4. 移植前必须先判断的事情
在移植之前，先回答下面几个问题：

1. 目标项目是不是也是 STM32？
2. 目标项目是否也采用 `CubeMX + CMake`？
3. 目标项目是否使用 `FreeRTOS`？
4. 目标项目的外设功能和当前工程重合到什么程度？
5. 复用目标到底是“底层能力”，还是“完整控制链路”？

如果只是想复用底层能力，优先带走：

1. `Common`
2. `Driver`
3. 部分 `Module`

如果还想复用完整链路，才考虑继续移植：

1. `Task`
2. 绑定表
3. 状态机

## 5. 推荐的移植顺序
### 第一步：先移植 Common

先把这些带过去：

1. `common_checksum`
2. `common_str`
3. `PID`

原因是：

1. 风险最低
2. 基本不依赖外设初始化
3. 能先建立公共算法层

### 第二步：移植 Driver

建议顺序：

1. `drv_gpio`
2. `drv_adc`
3. `drv_pwm`
4. `drv_encoder`
5. `drv_key`
6. `drv_uart`

移植 Driver 时需要重点确认：

1. 目标项目是否已经生成了对应 `HAL` 句柄
2. `drv_uart` 支持的 UART 实例映射是否和目标项目一致
3. 定时器模式是否已经在 `.ioc` 中正确配置

### 第三步：移植 Module

建议顺序：

1. `mod_led`
2. `mod_relay`
3. `mod_battery`
4. `mod_key`
5. `mod_sensor`
6. `mod_motor`
7. `mod_uart_guard`
8. `mod_vofa`
9. `mod_k230`
10. `mod_stepper`

原因是：

1. 先移植简单的 GPIO/ADC 语义模块，更容易验证基础环境。
2. 再移植电机和协议模块，方便分阶段排错。

### 第四步：最后才考虑移植 Task

`Task` 层耦合最强。  
只有当目标项目确实希望复用当前业务流程时，才建议一并移植。

## 6. 移植时每一层分别要改什么
### 6.1 移植 Common

一般只需要：

1. 拷贝文件
2. 加入构建系统
3. 确认头文件路径

### 6.2 移植 Driver

需要重点处理：

1. HAL 头文件路径
2. `main.h`、`usart.h`、`tim.h`、`adc.h` 等文件位置
3. 新工程实际启用的 UART 实例
4. 新工程的 TIM 通道和编码器模式

### 6.3 移植 Module

需要重点处理：

1. 硬件绑定表
2. 默认上下文的装配方式
3. 新项目中哪些模块需要保留、哪些不需要

### 6.4 移植 Task

需要重点处理：

1. RTOS 线程创建
2. 资源创建
3. 初始化闸门逻辑
4. 当前工程任务状态机是否适合新项目

## 7. 当前工程中最具复用价值的设计
### 7.1 上下文 `ctx` 设计

大量模块采用：

1. `get_default_ctx`
2. `ctx_init`
3. `bind`
4. `is_bound`
5. 运行期 API

这种设计的好处是：

1. 生命周期清楚
2. 状态集中
3. 容易做默认实例
4. 容易做多实例扩展

### 7.2 绑定表设计

像 `mod_led`、`mod_relay`、`mod_motor`、`mod_sensor`、`mod_key` 都通过绑定表注入硬件映射。  
这意味着换板时，不是去全项目搜索 GPIO，而是集中修改绑定表。

### 7.3 串口仲裁设计

`mod_uart_guard` 是当前工程里非常值得保留的设计。  
它解决的是：

1. 多个协议模块可能抢同一个 UART
2. 同一 owner 可能重入 claim
3. 绑定和解绑时需要防止误释放其他模块资源

如果目标项目有多个串口协议模块并存，这套设计非常有价值。

## 8. 针对不同复用目标的建议
### 场景 1：只想复用电机控制基础能力

建议带走：

1. `drv_gpio`
2. `drv_pwm`
3. `drv_encoder`
4. `mod_motor`
5. `PID`

### 场景 2：只想复用灰度循迹输入

建议带走：

1. `drv_gpio`
2. `mod_sensor`

然后只改：

1. 通道数
2. 引脚映射
3. 权重表

### 场景 3：只想复用串口协议能力

建议带走：

1. `drv_uart`
2. `mod_uart_guard`
3. `mod_vofa`
4. `mod_k230`
5. `mod_stepper`

### 场景 4：想复用整套结构

建议顺序：

1. 先在新项目中重建 `.ioc`
2. 再接入构建系统
3. 再接入 `Common/Driver/Module`
4. 最后按需移植 `Task`

## 9. 把这套库移植到新板子时的标准步骤
### 第一步：先重建底层外设

在新板子的 `.ioc` 中先把这些配出来：

1. 需要的 GPIO
2. 需要的 PWM 定时器
3. 需要的编码器定时器
4. 需要的 ADC
5. 需要的 UART
6. 如果有 OLED，再配 I2C

### 第二步：确认 HAL 句柄名

因为模块绑定通常需要：

1. `&huartX`
2. `&htimX`
3. `&hadcX`
4. `GPIOx`
5. `GPIO_PIN_x`

因此必须确认新工程句柄名称和当前工程能否对应。

### 第三步：重写绑定表

这是移植里最关键的一步。  
大多数“编译能过但运行不对”的问题都出在这里。

通常需要改的不是模块源码，而是：

1. LED 绑定表
2. Relay 绑定表
3. Key 绑定表
4. Motor 绑定表
5. Sensor 绑定表
6. Battery 绑定参数
7. UART 协议模块绑定

### 第四步：做最小可运行验证

建议按这个顺序：

1. 先验 GPIO
2. 再验 ADC
3. 再验 UART
4. 再验 PWM
5. 再验 Encoder
6. 最后再上闭环控制

## 10. 如何扩展新的模块
### 10.1 新模块建议采用同样的结构

建议至少包含：

1. `mod_xxx_bind_t`
2. `mod_xxx_ctx_t`
3. `mod_xxx_get_default_ctx`
4. `mod_xxx_ctx_init`
5. `mod_xxx_bind`
6. `mod_xxx_is_bound`
7. 运行期接口

### 10.2 新模块该放在哪层

判断规则很简单：

1. 如果只是最小硬件访问，放 `Driver`
2. 如果是设备语义或协议封装，放 `Module`
3. 如果是状态机和时序，放 `Task`
4. 如果是纯算法工具，放 `Common`

## 11. 如何扩展新的任务
新增任务前先问自己：

1. 这件事是不是必须独立调度？
2. 它是不是会和现有任务争抢资源？
3. 它应该消费哪些模块？
4. 它应该受不受 `InitTask` 闸门控制？

一般新增任务时，至少要同步修改：

1. `freertos.c`
2. 新的 `task_xxx.h/.c`
3. 相关同步资源
4. 文档

## 12. 当前移植时最容易踩的坑
### 坑 1：只复制源码，不复制构建配置

这会导致文件虽然在仓库里，但并没有参与编译。

### 坑 2：只改 `.ioc`，不改绑定表

这在当前工程结构里是最典型的错误。

### 坑 3：串口实例没对齐

尤其是 `drv_uart`、`mod_uart_guard`、`mod_k230`、`mod_stepper` 这几块，一旦串口实例错了，协议链路一定不通。

### 坑 4：方向定义没有统一

需要同时确认：

1. 电机方向
2. 编码器方向
3. 传感器左右顺序
4. 步进正反方向

### 坑 5：把项目层逻辑误当成通用库

例如当前 `DccTask` 的状态机设计，不一定适用于下一个目标项目。  
不要因为它现在能跑，就把它当成不可修改的标准模板。

## 13. 推荐的复用思路总结
如果计划长期把这套工程当成“通用驱动库母工程”，最合理的维护思路是：

1. 保持 `Common/Driver/Module` 的接口尽量稳定
2. 把具体项目差异留在 `Task` 和绑定表
3. 引脚变化优先收敛到 `.ioc + task_init.c`
4. 协议变化优先收敛到对应 `Module`
5. 每次扩展后同步更新文档

## 14. 什么时候必须更新这份文档
只要下面内容变了，就要更新本文件：

1. 哪些目录适合复用的判断变了
2. 模块生命周期风格变了
3. 串口资源仲裁模型变了
4. 典型移植步骤变了
5. 推荐的扩展规范变了
