# 系统启动流程与运行逻辑总览
## 1. 这份文档给谁看
这份文档主要给 3 类人看：

1. 后续重新阅读工程时，可快速重新建立整体认知。
2. 后续接手维护的人，需要在不翻完整个仓库的前提下看懂系统主链路。
3. 毕设答辩或项目展示时，需要清楚说明“从底层初始化到业务运行到底发生了什么”。

这份文档不重点讲安装软件，也不重点讲具体引脚。  
它重点回答的是：

1. MCU 上电后，代码先干了什么。
2. FreeRTOS 里到底创建了哪些任务和资源。
3. 任务之间怎么协作。
4. 底盘、循迹、步进、显示、按键反馈这几条链路是怎么串起来的。

## 2. 系统分层的核心思想
本工程采用的核心分层关系是：

`Task -> Module -> Driver -> HAL`

这 4 层分别负责：

1. `Task`
   - 调度
   - 状态机
   - 时序
   - 多模块协作
2. `Module`
   - 设备语义
   - 协议封装
   - 绑定映射
   - 上下文生命周期
3. `Driver`
   - GPIO
   - PWM
   - Encoder
   - UART
   - ADC
   - Key 等最小硬件访问能力
4. `HAL/Core`
   - 由 `CubeMX` 生成的底层初始化

这套结构的核心价值并不在于“形式上规范”，而在于：

1. 换引脚时，不必把业务逻辑推翻重写。
2. 换任务逻辑时，不必直接碰 HAL。
3. 模块能抽出来复用到别的项目。

## 3. 上电后的完整启动顺序
### 3.1 硬件上电后

MCU 上电后会先进入启动文件和系统时钟初始化，然后进入 `main()`。

### 3.2 `main()` 里当前的初始化顺序
根据当前工程，`main()` 中会依次执行：

1. `HAL_Init()`
2. 时钟配置
3. `MX_GPIO_Init()`
4. `MX_DMA_Init()`
5. `MX_USART3_UART_Init()`
6. `MX_ADC1_Init()`
7. `MX_USART2_UART_Init()`
8. `MX_TIM4_Init()`
9. `MX_UART5_Init()`
10. `MX_TIM2_Init()`
11. `MX_TIM3_Init()`
12. `MX_I2C2_Init()`
13. `MX_UART4_Init()`
14. `MX_FREERTOS_Init()`
15. 启动 RTOS 调度器

需要注意两点：

1. 这里做的是“底层外设初始化”，不是业务初始化。
2. 真正的模块绑定和安全态设置，不在 `main.c`，而在 `InitTask`。

## 4. FreeRTOS 资源与任务创建
### 4.1 当前创建的任务

| 任务名 | 入口函数 | 优先级 | 栈大小 | 作用 |
| --- | --- | --- | --- | --- |
| `defaultTask` | `StartDefaultTask` | `Low` | `128*4` | 保底空转任务 |
| `GpioTask` | `StartGpioTask` | `Low` | `128*4` | LED/蜂鸣器/激光反馈 |
| `KeyTask` | `StartKeyTask` | `Low` | `128*4` | 扫描按键并分发事件 |
| `OledTask` | `StartOledTask` | `Low` | `128*4` | 电压和状态显示 |
| `TestTask` | `StartTestTask` | `High` | `128*4` | 测试预留 |
| `StepperTask` | `StartStepperTask` | `Realtime` | `1024*4` | 双轴步进控制 |
| `DccTask` | `StartDccTask` | `Realtime` | `512*4` | 底盘状态机与控制 |
| `InitTask` | `StartInitTask` | `Realtime7` | `256*4` | 模块绑定与一次性初始化 |

### 4.2 当前创建的同步资源

| 资源名 | 类型 | 作用 |
| --- | --- | --- |
| `Sem_InitHandle` | 信号量 | 初始化闸门 |
| `Sem_RedLEDHandle` | 信号量 | 任意有效按键后的黄灯/短鸣反馈 |
| `Sem_DccHandle` | 信号量 | KEY2 单击触发 DCC 复位 |
| `Sem_TaskChangeHandle` | 信号量 | KEY3 单击切换模式 |
| `Sem_ReadyToggleHandle` | 信号量 | KEY3 双击切换运行态 |
| `PcMutexHandle` | 互斥锁 | 串口发送互斥，主要给 VOFA/Stepper 使用 |

## 5. InitTask 为什么是整个系统的关键
### 5.1 它做了什么

`InitTask` 是这套工程真正的“系统装配层”。  
它不是普通任务，而是一次性完成下面几件大事：

1. 绑定 LED 映射
2. 绑定继电器映射
3. 绑定按键映射
4. 绑定双电机映射
5. 绑定 12 路灰度传感器映射
6. 绑定电池采样参数
7. 初始化上述基础模块
8. 绑定 VOFA 串口
9. 绑定 K230 串口
10. 准备双轴步进通道
11. 释放初始化闸门
12. 自删除

### 5.2 为什么要这样设计
如果没有 `InitTask`，通常会出现下面这些问题：

1. 某个任务启动得早，结果先访问了还没绑定的模块。
2. 不同任务各自做自己的初始化，导致装配逻辑分散、难维护。
3. 后续换引脚、换串口时，不知道该去哪改统一入口。

现在的做法是：

1. 硬件底层初始化交给 `CubeMX/Core`
2. 模块装配交给 `InitTask`
3. 业务运行交给其它任务

这也是当前工程中最值得保留的结构之一。

## 6. 初始化闸门机制
### 6.1 机制本身

系统里存在一个重要信号量：`Sem_InitHandle`。  
它的作用不是普通事件通知，而是“启动闸门”。

所有业务任务在进入主循环前，都会调用：

`task_wait_init_done()`

这个函数的逻辑是：

1. 等 `InitTask` 释放一次信号量。
2. 第一个通过的任务会再把信号量放回去。
3. 后续任务也都能通过。

### 6.2 这样做的好处

1. 不需要每个任务都自己判断“模块有没有初始化好”。
2. 所有任务都能在逻辑上等待同一个启动完成点。
3. `InitTask` 只负责一次性装配，完成后退出。

## 7. 各条业务链路怎么协作
### 7.1 按键链路

完整链路如下：

`按键硬件 -> drv_key -> mod_key -> KeyTask -> 信号量 -> DccTask/GpioTask`

具体过程：

1. `KeyTask` 每 `10ms` 扫描一次按键。
2. `mod_key` 内部调用 `drv_key` 做去抖、单双击、长按识别。
3. 任何有效事件都会先释放 `Sem_RedLEDHandle`。
4. `KEY2` 单击释放 `Sem_DccHandle`
5. `KEY3` 单击释放 `Sem_TaskChangeHandle`
6. `KEY3` 双击释放 `Sem_ReadyToggleHandle`

这条链路体现了当前工程的一个重要原则：  
按键任务不直接改 DCC 状态，而是只发信号。

### 7.2 GPIO 反馈链路

完整链路如下：

`KeyTask/DccTask 状态 -> GpioTask -> mod_led/mod_relay -> drv_gpio`

`GpioTask` 每 `20ms` 轮询一次，根据当前 DCC 运行状态和按键反馈窗口输出：

1. 黄灯短闪
2. 按键短鸣
3. ON 态绿灯闪烁
4. STOP 态红灯闪烁
5. STOP 态节奏蜂鸣
6. ON 态激光继电器吸合

这意味着：

1. 不要在别的任务里直接写 LED。
2. 所有反馈输出统一由 `GpioTask` 仲裁，才不会相互打架。

### 7.3 OLED 显示链路

完整链路如下：

`mod_battery + task_dcc 状态 -> OledTask -> mod_oled -> I2C2`

`OledTask` 做两件分开的事：

1. 低频采样电池电压
2. 周期刷新页面

当前页面主要显示：

1. `voltage`
2. `mode`
3. `task:OFF/PREP/ON/STOP`

### 7.4 底盘 DCC 链路

完整链路如下：

`按键信号量 -> DccTask -> mod_sensor/mod_motor/PID -> 双轮输出`

`DccTask` 是底盘控制状态机唯一写入者。  
它维护 2 组状态：

1. 模式：
   - `IDLE`
   - `STRAIGHT`
   - `TRACK`
2. 运行态：
   - `OFF`
   - `PREPARE`
   - `ON`
   - `STOP`

#### 模式和运行态为什么分开
这是非常关键的一点。

1. 模式回答的是“运行时按哪种策略控制”
2. 运行态回答的是“现在允不允许输出”

所以才会出现：

1. 当前模式是 `TRACK`
2. 但运行态仍然是 `OFF`

这两者并不矛盾。

#### DCC 的典型运行过程

1. 上电后默认 `IDLE + OFF`
2. KEY3 单击循环切换模式
3. KEY3 双击切换运行态
4. 从 `OFF` 进入 `PREPARE`
5. `PREPARE` 倒计时完成后进入 `ON`
6. `ON` 下按当前模式执行控制
7. 遇到保护条件进入 `STOP`
8. `STOP` 下强制回 `IDLE`

### 7.5 STRAIGHT 模式

在 `STRAIGHT` 模式下，当前工程实现的是：

1. 位置外环
2. 左右速度内环
3. 输出左右轮占空比

同时它带了两个重要保护：

1. 上电启动额外延时 `3000ms`
2. 进入直线模式后的前 `1000ms` 传感器保护窗

之后如果连续检测到黑线达到阈值，就会进入 `STOP`。

### 7.6 TRACK 模式

在 `TRACK` 模式下：

1. 12 路灰度传感器先计算权重偏差
2. `DccTask` 根据偏差修正左右轮目标速度
3. 最终仍通过 `mod_motor + 编码器反馈 + PID` 去闭环输出

因此需要明确：

1. 灰度模块不是直接驱动电机
2. 它只是提供控制偏差输入

### 7.7 步进链路

完整链路如下：

`K230 -> mod_k230 -> StepperTask -> mod_stepper -> drv_uart -> 双轴步进驱动`

当前 `StepperTask` 的特征非常明确：

1. 周期 `10ms`
2. 按 `K230` 最新误差帧进行双轴控制
3. X 轴使用 `UART5`
4. Y 轴使用 `USART2`
5. 内置软限位
6. 容忍短时丢帧
7. 连续无新帧达到阈值会停机保护

此外它还做了一个很有价值的设计：

1. 在 `STRAIGHT` 模式下，可以结合底盘编码器差分做前馈
2. 这说明步进控制并不是完全孤立于底盘状态的

## 8. 当前系统的时序参数
### 8.1 主要任务周期

| 任务 | 周期 |

| `TASK_STEPPER_Y_LIMIT_PULSE` | `400` | Y 轴软限位 |

## 9. 典型数据流理解方式
### 9.1 排查“为什么模式切换了”

按这个顺序看：

1. `KeyTask`
2. `Sem_TaskChangeHandle`
3. `DccTask` 状态机
4. `OledTask` 和 `GpioTask` 的显示/反馈

### 9.2 排查“为什么电机不转”

按这个顺序看：

1. 当前 `DCC` 运行态是不是 `ON`
2. 模式是不是允许输出
3. `mod_motor` 有没有初始化成功
4. PWM 和编码器绑定对不对
5. 方向修正是不是弄反了

### 9.3 排查“为什么步进不动”

按这个顺序看：

1. `K230` 有没有新帧
2. `StepperTask` 是否 ready
3. 对应轴是否启用
4. 误差是否落入死区
5. 是否撞软限位
6. UART 和驱动是否共地

## 10. 为什么这套结构适合继续扩展
这套结构可扩展，不是因为功能多，而是因为职责边界清楚：

1. 新增一个显示页面，不需要动底层 UART
2. 新增一个协议模块，不需要改 DCC 状态机
3. 换一个引脚，不需要改 PID 算法
4. 想把驱动库移植走，可以优先带走 `Common/Driver/Module`

## 11. 维护者最容易犯的错误
### 错误 1：在任务里直接写 HAL

这会直接破坏当前工程最重要的分层价值。

### 错误 2：把业务初始化散落到多个任务

这会削弱 `InitTask` 作为统一装配入口的意义。

### 错误 3：改了 `.ioc` 但没改 `task_init.c`

这样代码能编译，但运行行为一定会错。

### 错误 4：看到功能异常就直接改 PID

很多问题其实是：

1. 绑定错了
2. 方向反了
3. 串口没通
4. 任务状态没到 `ON`

不是 PID 本身的问题。

## 13. 这份文档在什么时候必须更新
只要下面任一项变化，就必须更新本文件：

1. 任务数量、优先级、栈大小变化

3. `InitTask` 绑定流程变化
