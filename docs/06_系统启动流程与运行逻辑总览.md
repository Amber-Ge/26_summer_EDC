# 系统启动流程与运行逻辑总览

## 1. 文档目的

本文档用于说明：

1. 当前工程的主架构
2. 上电后到 RTOS 启动的主链路
3. `InitTask` 在当前版本中的装配职责
4. 哪些任务当前已接入默认启动流程，哪些只是保留资产

## 2. 当前主架构

当前工程仍采用：

`Task -> Module -> Driver -> HAL`

各层职责简述如下：

1. Task：任务调度、状态机、时序和多模块协作
2. Module：设备语义、协议、上下文、绑定结构
3. Driver：最小硬件访问封装
4. HAL/Core：CubeMX 生成的底层初始化

本次整理没有引入新的 Board/Bsp 启动架构，也没有推翻 `InitTask` 作为装配入口的现有设计。

## 3. 上电到 RTOS 启动的主链路

当前主流程仍然是：

1. MCU 上电
2. 启动文件与时钟初始化
3. 进入 `main()`
4. 执行 `MX_GPIO_Init()`、`MX_DMA_Init()`、`MX_USART*_*Init()`、`MX_TIM*_*Init()`、`MX_I2C*_*Init()` 等 CubeMX 生成初始化
5. 执行 `MX_FREERTOS_Init()`
6. 启动调度器

需要强调：

1. `main.c` 负责底层外设初始化
2. `InitTask` 负责上层装配和绑定注入
3. 本次未修改 CubeMX 生成区与 HAL 初始化风格

## 4. 当前默认创建的任务

当前默认启动链以 [freertos.c](/D:/AAAAAA26电赛/Summer_26_zgt6_freertos/Core/Src/freertos.c) 为准。

当前实际创建的线程如下：

| 任务名 | 入口函数 | 作用 |
| --- | --- | --- |
| `defaultTask` | `StartDefaultTask` | 保底空转任务 |
| `InitTask` | `StartInitTask` | 模块绑定与一次性初始化 |

当前仓库中仍保留以下任务实现，作为工程资产继续维护：

1. `task_gpio`
2. `task_key`
3. `task_oled`
4. `task_dcc`
5. `task_stepper`

它们当前是否启用，不以文件是否存在为准，而以 `freertos.c` 是否创建对应线程为准。

测试预留任务 `task_test` 已归档到 `99_Archive`，默认不参与编译。

## 5. InitTask 的当前作用

`InitTask` 在当前版本中仍然承担装配中心角色，负责：

1. 默认模块上下文绑定
2. LED / Relay / Key / Motor / Sensor / Battery 等绑定表注入
3. 基础模块初始化与安全态设置
4. VOFA / K230 / Stepper 通道准备
5. 释放初始化闸门

这是一种当前工程语境下合理且稳定的做法，本次保持不变。

## 6. 启动闸门机制

当前仍通过 `task_wait_init_done()` 保证：

1. 业务任务不会抢跑
2. 先完成装配，再进入业务循环

这套机制与 `InitTask` 相配合，是当前工程稳定性的关键组成部分，本次未拆散。
